## Prose Examples (Step 1A: Narrative Draft)

Below are examples showing **three fundamentally different rhetorical strategies**. Each example also shows what a BAD (단순 나열) version looks like vs. the GOOD (사고가 담긴) version.

```
Example A — 논증형: 왜 이 접근법이 필요한가 (현대차 MDPS 열화 진단)
Rhetorical strategy: 문제의 심각성 → 기존 해법의 한계 → 우리 접근법의 논리적 필연성
──────────────────────────────────────────────────────────────────────

❌ BAD (단순 나열):
  기계 건강 상태 진단에서 운전 조건 변동이 문제이다. 풍력 터빈 연구에서
  운전 영역별 안정성이 10배 차이났다. MDPS에서도 토크 변동이 있다.
  GMM을 사용하여 맥락 군집화를 할 것이다.
  → 사실만 나열. 왜 기존 방식이 안 되는지, 왜 GMM이어야 하는지 논리가 없음.

✅ GOOD (논증 구조):
  기계 건강 상태를 진단할 때 핵심적 난점은, 열화에 의한 신호 변화보다
  운전 조건에 의한 신호 변화가 훨씬 크다는 것이다. Lopez de Calle (2019)의
  풍력 터빈 연구에서, 동일한 기어박스라도 운전 영역에 따라 건강 지표의
  안정성이 최대 10배까지 차이났고, 5개 영역 중 오직 Pre-ramp에서만
  실제 열화 추세가 관측 가능했다. 나머지 4개 영역에서는 운전 변동의 잡음이
  열화 신호를 완전히 압도했다.

  MDPS에서도 근본적으로 같은 문제가 존재한다. 차량 속도, 노면 조건, 조향
  패턴이 복합적으로 작용하여 토크 특성을 교란한다. 1차년도에 시도한 수동
  규칙(속도·조향각 필터링)은 1-2차원의 교란만 통제 가능했으나, 실제 교란은
  5차원 이상이므로 근본적으로 불충분했다. 이것이 데이터 기반 자동 맥락
  군집화(GMM)가 수동 규칙을 대체해야 하는 이유이다.
  → "문제가 있다"가 아니라 "왜 기존 해법이 실패하고, 왜 이 해법이 필연적인가"를 논증
```

```
Example B — 의사결정형: 왜 이렇게 설계했는가 (LG전자 RMS 데이터 가공)
Rhetorical strategy: 설계 선택의 이유 → 대안과의 비교 → 선택의 결과
──────────────────────────────────────────────────────────────────────

❌ BAD (단순 나열):
  RMS 데이터는 6축 로봇의 위치/속도/토크/온도 + Job 데이터이다.
  Sampling rate는 10Hz이다. 접근(1.2s, 17,519개), 이송(4.0s, 32,908개),
  복귀(3.8s, 32,909개)로 분할했다.
  → 스펙 나열. 왜 이 분할 방식인지, 다른 선택지는 없었는지 사고가 없음.

✅ GOOD (설계 근거 포함):
  로봇 이상탐지에서 가장 먼저 해결해야 할 문제는 "무엇을 하나의 관측
  단위로 볼 것인가"이다. 연속 시계열을 그대로 쓰면 정상/이상 구간의
  경계가 모호해지고, 고정 길이 윈도우로 자르면 동작 중간에 끊기는
  문제가 생다. 우리는 로봇의 실제 작업 의미에 기반한 분할을 택했다.

  RMS 데이터의 Job_name 열에는 로봇의 현재 작업명이 기록되어 있다.
  이 값의 변동 시점을 경계로, 접근(타겟 이동, 1.2s), 이송(클램핑 후
  이동, 4.0s), 복귀(원점 복귀, 3.8s) 세 가지 단위 동작으로 분할했다.
  이 분할의 핵심 장점은, 같은 "이송" 동작끼리 비교하면 로봇 경로와
  하중이 거의 동일하므로 이상에 의한 토크 변화만 남는다는 것이다.
  만약 접근과 이송을 구분하지 않으면, 하중 유무에 따른 토크 차이가
  이상 신호보다 훨씬 커서 탐지가 불가능해진다.
  → "무엇을 했다"가 아니라 "왜 이것을 선택했고, 안 했으면 어떻게 됐을까"를 설명
```

```
Example C — 인사이트형: 결과가 의미하는 것 (LG전자 MCD 이상탐지 성능)
Rhetorical strategy: 기대 → 실제 결과 → 해석 → 한계 인정
──────────────────────────────────────────────────────────────────────

❌ BAD (단순 나열):
  Precision 94.2%, Recall 89.7%, F1-Score 91.9%를 달성했다.
  Block Size 256을 사용했다. FPR은 2.1%이다.
  공구 마모 초기 Recall은 78.3%로 낮다.
  → 숫자 나열. 이 숫자들이 무엇을 의미하는지, 왜 중요한지 해석이 없음.

✅ GOOD (해석 포함):
  MCD 모델의 전체 F1-Score는 91.9%로, 프로토타입 목표(≥90%)를
  달성했다. 특히 FPR 2.1%는 공장 현장에서 중요한 의미를 가진다 —
  하루 평균 500회 판정 기준으로 오탐은 약 10건 수준이며, 작업자가
  수용 가능한 범위이다.

  그러나 이상 유형별로 성능 차이가 뚜렷하다. 충돌/비정상 정지처럼
  토크 잔차가 급격히 변하는 이상은 Recall 96% 이상이지만, 공구 마모
  초기 단계는 78.3%에 그쳤다. 이는 마모 초기의 진동 변화가 정상
  변동 범위 내에 있어 Mahalanobis Distance가 Threshold를 넘지
  못하기 때문이다. Block Size를 256에서 512로 늘리면 F1이 93.1%로
  오르지만 Latency가 0.8ms→1.6ms로 두 배 증가하여, 실시간 요건
  (< 2ms)과의 trade-off가 발생한다. 현재로서는 Block Size 256이
  정확도와 실시간성의 최적 균형점이다.
  → "수치가 무엇을 의미하고, 왜 그런 결과가 나왔고, 어떤 trade-off가 있는지" 해석
```

## Compressed Examples (Step 1B: Bullet Compression)

Notice how each GOOD example has a **different rhetorical structure**:
- A = argument chain (문제 → 기존 해법 실패 → 새 접근 필연성)
- B = design rationale (선택지 → 선택 이유 → 안 했으면 어떻게 됐을까)
- C = interpretation (기대 → 결과 → 해석 → 한계)

Your slides should mix these strategies. A "배경" slide uses strategy A. A "방법론" slide uses strategy B. A "결과" slide uses strategy C. Never use the same strategy for all slides.

```
Example A — 논증형 compressed:
──────────────────────────────
– 운전 조건 변동이 열화 신호를 압도하는 교란 원인으로 작용
  → 풍력 터빈 3기, 6개월 모니터링 → 운전 영역별 건강 지표 안정성 최대 10배 차이
  → 5개 영역 중 Pre-ramp에서만 실제 열화 추세 관측 가능, 나머지 4개 영역 진단 불가
– 1차년도 수동 규칙(속도·조향각 필터링) → 1~2차원 교란만 통제 가능
  → 실제 교란은 5차원 이상 → 수동 규칙 기반 맥락 통제의 근본적 한계
– 5D 물리 특징 공간에서 GMM 기반 자동 맥락 군집화 → 다차원 교란 통제
  → 차종별 수동 규칙 정의 불필요 → Self-Evolving AI 컨셉의 필수 Contextualization 절차
```

```
Example B — 의사결정형 compressed:
──────────────────────────────────
– "무엇을 하나의 관측 단위로 볼 것인가" → 분할 방식이 이상탐지 성능 결정
  → 연속 시계열 그대로 사용 → 정상/이상 경계 모호
  → 고정 길이 윈도우 → 동작 중간에 절단, 맥락 상실
– 로봇 작업 의미 기반 분할 → Job_name 값 변동 시점을 경계로 3종 단위 동작
  → 접근(1.2s, 17,519개) / 이송(4.0s, 32,908개) / 복귀(3.8s, 32,909개)
– 동일 "이송" 끼리 비교 시 경로·하중 거의 동일 → 이상에 의한 토크 변화만 잔류
  → 접근/이송 미구분 시 하중 유무 토크 차이가 이상 신호 압도 → 탐지 불가
```

```
Example C — 인사이트형 compressed:
──────────────────────────────────
– F1-Score 91.9% → 프로토타입 목표(≥90%) 달성
  → FPR 2.1% → 하루 500회 판정 기준 오탐 약 10건 → 작업자 수용 가능 수준
– 충돌/비정상 정지(급격한 토크 잔차) → Recall 96% 이상
  → 공구 마모 초기 → Recall 78.3%에 그침
  → 마모 초기 진동 변화가 정상 변동 범위 내 → Mahalanobis Distance가 Threshold 미초과
– Block Size 256→512 시 F1 91.9%→93.1% 향상, Latency 0.8→1.6ms 증가
  → 실시간 요건(< 2ms)과의 균형 → Block Size 256이 현재 최적 균형점
– 공구 마모 초기 탐지 Recall 개선 필요 → Feature Engineering 고도화 또는 DL 모델 검토
```

## Visual Specification Examples (Step 1C)

```
Example A — 논증형 슬라이드의 시각 자료:
────────────────────────────────────────
Visual: 2-panel comparison diagram (left: 수동 규칙, right: GMM 기반)
  Left: 속도·조향각 → 1차원 필터 → 잔여 교란 큼 (❌)
  Right: 5D 물리 특징 → GMM 군집화 → 맥락 내 열화 분리 (✅)

Self-check:
  ✓ 텍스트는 "수동 규칙이 근본적으로 불충분"이라 주장 → 그림이 '왜' 불충분한지 보여줌
  ✓ 좌우 대비 → 3초 만에 "1차원 vs 5차원" 접근법 차이 파악
  ✗ 텍스트를 단순 반복하는가? → No, 그림이 차원 축소/군집 시각화를 추가로 보여줌
  → PASS
```

```
Example B — 의사결정형 슬라이드의 시각 자료:
─────────────────────────────────────────────
Visual: 시계열 차트 (position 데이터)
  접근/이송/복귀 구간을 3가지 색상으로 구분 표시
  + 구간 경계에 Job_name 변동 시점 마커 (수직 점선)

Self-check:
  ✓ 텍스트는 "왜 의미 기반 분할을 택했는가"를 논증 → 차트는 실제 분할 결과를 보여줌
  ✓ 색상 구분으로 3초 만에 "동작이 규칙적으로 반복됨" 파악 가능
  ✗ 텍스트를 단순 반복하는가? → No, 실제 데이터 파형은 텍스트에 없는 정보
  ✗ 테이블로 대체 가능한가? → 테이블은 숫자만 보여줌, 시계열 패턴은 차트만 가능
  → PASS
```

```
Example C — 인사이트형 슬라이드의 시각 자료:
─────────────────────────────────────────────
Visual 1: Bar chart — 이상 유형별 Recall 비교
  충돌(96.2%), 비정상 정지(94.8%), 공구 마모 중기(87.1%), 공구 마모 초기(78.3%)
  → 마모 초기만 유독 낮다는 것이 한 눈에 드러남
Visual 2: Table — Block Size vs F1/Latency trade-off
  64/128/256(채택)/512/1024 행, F1-Score/Latency/판정 열

Self-check:
  ✓ 차트: 텍스트의 "유형별 차이가 뚜렷"을 시각적으로 입증 → 어디가 약한지 즉시 파악
  ✓ 테이블: Block Size 선택 근거를 한 눈에 비교 → 텍스트의 trade-off 논의를 뒷받침
  ✗ 차트가 텍스트 반복인가? → No, 4개 유형의 상대적 크기는 차트만 가능
  → PASS
```
